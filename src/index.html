Craby — Terminal Native AI Coding Agent

An agentic, terminal based coding assistant that plans, edits across as many files as a task needs, runs your tests, and remembers your project between sessions with every change staged and shown to you as a diff before anything touches disk.

Craby

Craby is a terminal-native AI coding agent written in TypeScript on Bun. No editor plugin, no IDE required — it works wherever you already have a terminal, against any model OpenRouter hosts or a fully local Ollama model with no cloud dependency at all. Nothing is written to your filesystem without explicit approval: staged diffs, batch or per-file review, and destructive-command warnings are the core safety model, not a bolt-on setting.

| Github: https://github.com/arijiiiiitttt/craby

Overview

Craby reads your codebase, stages file changes, then asks for your approval before writing anything. It supports three modes — Agent, Ask, and Plan — and gets better the more you use it on a given project: durable facts, conventions, and decisions persist across sessions via craby memory, so you stop re-explaining the same context every time.

Every applied session is recorded in an undo journal (independent of git), so craby undo can precisely revert just what craby changed. It runs on macOS, Linux, and Windows anywhere Bun runs, and can be pointed at a local Ollama model to skip the cloud entirely.

Features

Coding Agent

  • Agent, Ask, and Plan modes for coding, read-only Q&A, and step-by-step planning
  • Staged diffs — nothing touches disk until you approve it, file by file or all at once
  • Multi-turn, streaming output that prints live as the model generates it
  • Self-correction loop that re-checks its own work after real tool calls, up to 2 rounds
  • Destructive-command warnings on any staged shell command
  • Test-aware — auto-detects and runs npm/yarn/pnpm/bun, pytest, cargo, or go tests against staged changes in an isolated sandbox
  • Semantic search over your codebase via a local, on-disk embedding index
  • craby memory — durable per-project facts and conventions that persist across sessions
  • craby undo — journal-based rollback of the last applied session, independent of git
  • Works against OpenRouter-hosted models or a fully local Ollama model

Project Structure

craby is organised into four packages plus a CLI entry point:

  • — packages/shared/ — config, memory, undo journal, prompts, risk detection
  • — packages/ai/ — model provider wiring for OpenRouter and Ollama
  • — packages/modes/agent/ — orchestrator, tool executor, approval flow, git and test tools
  • — packages/modes/ask/ — read-only codebase Q&A
  • — packages/modes/plan/ — read-only investigation, produces a numbered markdown plan
  • — packages/tui/ — terminal banner, menu, and interactive picker

How It Works :

craby is built in six layers. Each layer handles one specific job and passes results up to the next. When you run a command, it travels through these layers and comes back as a staged, reviewable change.

Layer 01

shared — Config, Memory & Safety

Holds the pieces every mode depends on: env/config loading, the per-project memory store, the undo journal, shared prompts, conversation state, and destructive-command risk detection. Memory is stored per-project under ~/.craby/memory/, not inside your repo, and is never sent anywhere except as context in your own requests to your configured model.

Layer 02

ai — Model Provider

Wires craby up to whichever model you've configured — any model OpenRouter hosts, or a fully local Ollama model with no cloud dependency. Streaming, token usage reporting, and embedding calls for semantic search all flow through this layer regardless of which provider is active.

Layer 03

modes/agent — Coding Agent Core

The brain of craby. The orchestrator reads your codebase, stages file creates/modifies/deletes in memory, and coordinates git tools, the test runner, and semantic search. Nothing is written to disk until the approval flow confirms it — a summary of changed files and +/- line stats is shown up front, with full diffs available per file.

Layer 04

modes/ask & modes/plan — Read-Only Modes

Ask mode is conversational Q&A over your codebase with no shell access and no file writes — only read_file, list_files, search_files, and analyze_codebase are available. Plan mode investigates read-only and produces a numbered markdown implementation plan, which can then be handed straight to Agent mode to execute.

Layer 05

tui — Terminal Interface

Renders the startup banner, the Agent / Ask / Plan / Help picker, and formats streamed output, diffs, and approval prompts directly in your terminal — no external client or editor plugin needed.

Layer 06

cli — Entry Point

Parses the command you ran (craby agent, craby ask, craby plan, craby undo, craby memory ..., craby --dir ...) and dispatches into the right mode, or straight into Agent mode when you pass a task string directly, e.g. craby "add input validation to the signup form".

Key Concepts Explained Simply

What is Staging?

Instead of writing files directly, craby holds every create, modify, and delete in memory first. The agent can even read back its own staged edits within the same session, so "make a change, then verify it" actually works. Nothing touches your real filesystem until you approve it all at once, or file by file with full diffs.

What is the Self-Correction Loop?

After a turn that made changes, craby automatically asks itself "is this actually done and correct?" and keeps fixing up to 2 rounds by default before handing control back to you. Pure conversational replies skip this; it only runs after real tool calls. It is like a second pair of eyes that checks its own work before showing it to you.

What is Memory?

Craby decides what's worth keeping about a project a stated preference, an architectural choice, a "never touch this folder" rule and saves it via the remember tool. The next session's system prompt includes it automatically, in Agent, Ask, and Plan mode alike, so you stop re-explaining the same context every time you open a session.

What is the Undo Journal?

Every applied session is recorded in an undo journal that is completely independent of git, so it works in any directory. Running craby undo precisely reverts just what craby changed, without touching any other edits you've made since. Think of it as a dedicated safety net separate from your normal version control.

What is Semantic Search?

If you configure an embedding model, semantic_search finds conceptually related code even when it doesn't share any keywords with your query, backed by a local on disk index. It's how craby finds the right file to edit in a large codebase without you having to name it.

What is AGENTS.md?

An AGENTS.md file in your project root gives craby persistent context stack, conventions, architecture decisions that gets injected into the system prompt automatically, in every mode. Craby also discovers and reads SKILL.md files the same way Claude Code and Cursor do.
